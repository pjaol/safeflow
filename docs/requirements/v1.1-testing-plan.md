# v1.1 Testing Plan — Accessibility & Internationalisation

**Status:** Active
**Branch:** `release/1.1`
**Date:** 2026-04-11

---

## Overview

v1.1 introduces no new user-facing features. The only risk it carries is regression — breaking something that already works while adding accessibility labels, replacing font sizes, or extracting strings into a catalog. The test strategy is therefore built around two goals:

1. **Prove the a11y and i18n work is actually correct** (not just merged)
2. **Catch regressions** before they reach TestFlight

All testing is automated except for one manual gate: a VoiceOver device walkthrough before submission. Everything else runs on every PR and on every merge to `release/1.1`.

## 1. Accessibility UI tests (`AccessibilityUITests.swift`)

**What it tests:** Every interactive element in the primary flows has an accessibility label. Every decorative image is hidden. The VoiceOver traversal order makes sense.

**How it works:** XCUITest can query elements by their accessibility label and assert on `.accessibilityLabel`, `.accessibilityHint`, and `.exists`. It can also enumerate all interactive elements in a container and assert none of them have an empty label.

### Test cases to implement

#### 1.1 No unlabelled interactive elements on home screen

```swift
func testHomeScreenHasNoUnlabelledButtons() {
    // All buttons must have a non-empty accessibility label
    let buttons = app.buttons.allElementsBoundByIndex
    for button in buttons {
        XCTAssertFalse(
            button.label.isEmpty,
            "Unlabelled button found: identifier=\(button.identifier)"
        )
    }
}
```

This catches any new icon-only button added without a label.

#### 1.2 Specific critical labels

Toolbar buttons are the highest-risk — they are image-only with no visible text. Assert each one by identifier and verify the label is a meaningful string (not empty, not the SF Symbol name):

```swift
func testToolbarButtonsHaveLabels() {
    XCTAssertEqual(app.buttons["home.settingsButton"].label, "Settings")
    XCTAssertEqual(app.buttons["home.editLogsButton"].label, "Edit logs")
    XCTAssertEqual(app.buttons["home.forecastButton"].label, "View forecast")
    XCTAssertEqual(app.buttons["home.supportButton"].label, "Get support")
}
```

#### 1.3 Decorative images do not appear in the accessibility tree

Images marked `.accessibilityHidden(true)` are invisible to XCUITest queries. Assert they don't appear:

```swift
func testDecorativeImagesAreHidden() {
    // These identifiers exist in the view hierarchy but should NOT be
    // queryable as accessibility elements
    XCTAssertFalse(app.images["decorative.settingsIcon"].exists)
    XCTAssertFalse(app.images["decorative.calendarIcon"].exists)
}
```

This requires giving decorative images a non-accessibility identifier for the assertion. Alternatively, assert that the *count* of accessibility-visible images on a screen is zero (all images are decorative):

```swift
func testHomeScreenHasNoAccessibilityImages() {
    // No images should be visible to VoiceOver on the home screen
    // (all images are decorative; information is conveyed by text/buttons)
    XCTAssertEqual(app.images.count, 0)
}
```

#### 1.4 Onboarding Stepper and DatePicker have labels

```swift
func testOnboardingInputsHaveAccessibilityLabels() {
    navigateToOnboardingCycleSetup()
    XCTAssertTrue(app.datePickers["onboarding.lastPeriodDatePicker"].exists)
    XCTAssertFalse(app.datePickers["onboarding.lastPeriodDatePicker"].label.isEmpty)
    XCTAssertFalse(app.steppers["onboarding.periodLengthStepper"].label.isEmpty)
    XCTAssertFalse(app.steppers["onboarding.cycleLengthStepper"].label.isEmpty)
}
```

#### 1.5 Cycle ring summary card count has context

The ring centre shows a bare number. After the fix, the element's accessibility label must include context:

```swift
func testRingCentreCountHasContext() {
    let ring = app.otherElements["home.cycleRingSummaryCard.centreCount"]
    XCTAssertTrue(ring.waitForExistence(timeout: 3))
    // Label must not just be a number — must include context word
    XCTAssertFalse(CharacterSet.decimalDigits.isSuperset(
        of: CharacterSet(charactersIn: ring.label)
    ), "Ring centre label is a bare number: \(ring.label)")
}
```

#### 1.6 All primary flows complete with no unlabelled elements (parameterised)

Run a helper that traverses every interactive element visible on screen and asserts non-empty labels. Call it after navigating to each primary screen:

```swift
func assertNoUnlabelledInteractiveElements(in app: XCUIApplication, screen: String) {
    for element in app.buttons.allElementsBoundByIndex
        + app.switches.allElementsBoundByIndex
        + app.steppers.allElementsBoundByIndex {
        XCTAssertFalse(
            element.label.isEmpty,
            "[\(screen)] Unlabelled element: type=\(element.elementType), id=\(element.identifier)"
        )
    }
}
```

## 2. Dynamic Type UI tests (`DynamicTypeUITests.swift`)

**What it tests:** At the three largest Dynamic Type sizes, no critical text is clipped, truncated mid-word, or obscured by an overflow layout.

**How it works:** XCUITest can launch the app with a `UIContentSizeCategory` environment override. Then assert that key elements exist, are on-screen, and have non-zero frame sizes.

### Test plan configuration

Dynamic Type tests run via a dedicated `.xctestplan` with multiple configurations (see §5). Each configuration sets a different `UIContentSizeCategory` override.

### Test cases to implement

#### 2.1 Home screen elements visible at AXXLarge

```swift
func testHomeScreenElementsVisibleAtMaxDynamicType() {
    // App launched with UIContentSizeCategory=AXXLarge via test plan config
    XCTAssertTrue(app.otherElements["home.cyclePhaseCard"].waitForExistence(timeout: 3))
    XCTAssertTrue(app.buttons["home.settingsButton"].exists)

    // Phase card text must not be zero-height (clipped)
    let phaseCard = app.otherElements["home.cyclePhaseCard"]
    XCTAssertGreaterThan(phaseCard.frame.height, 0)
}
```

#### 2.2 Log form inputs visible at AXXLarge

```swift
func testLogFormVisibleAtMaxDynamicType() {
    app.buttons["home.newLogButton"].tap()
    XCTAssertTrue(app.buttons["logDay.flow.medium"].waitForExistence(timeout: 3))
    XCTAssertTrue(app.buttons["logDay.saveButton"].exists)
    // Save button must be on screen (not scrolled off by large text)
    XCTAssertTrue(app.buttons["logDay.saveButton"].isHittable)
}
```

#### 2.3 Snapshot comparison (regression detection)

For pixel-level regression detection, generate reference screenshots at each Dynamic Type size during the first passing run. Subsequent runs diff against them. A layout shift of >5% of the screen height triggers a failure.

**Implementation:** Use Fastlane's existing `snapshot()` infrastructure extended with Dynamic Type launch environment:

```swift
func testSnapshotHomeAtLargeType() {
    // Set via test plan environment variable: UIContentSizeCategory=UICTContentSizeCategoryL
    snapshot("home-large-type")
}

func testSnapshotHomeAtAXXLargeType() {
    // Set via test plan: UIContentSizeCategory=UICTContentSizeCategoryAccessibilityExtraExtraLarge
    snapshot("home-axxlarge-type")
}
```

Reference screenshots are committed to `Tests/ReferenceScreenshots/` and diffed on CI.

## 3. Localisation UI tests (`LocalisationUITests.swift`)

**What it tests:** After string catalog extraction, the correct localised string appears in the UI for each language configuration. No raw string key appears in the UI (a missing translation shows the key name). Layout does not break under longer translations.

**How it works:** Tests run via `.xctestplan` configurations that set `AppleLanguages` and `AppleLocale`. The same test methods run for each language; the test plan handles the locale switching.

### Test cases to implement

#### 3.1 No raw string keys visible in UI

Raw keys look like `"settings.security.requireAuth"`. If a translation is missing, SwiftUI falls back to the key itself. Assert that no visible static text matches the key format:

```swift
func testNoRawStringKeysVisible() {
    let allText = app.staticTexts.allElementsBoundByIndex
        + app.buttons.allElementsBoundByIndex
    for element in allText {
        XCTAssertFalse(
            element.label.contains(".") && element.label.lowercased() == element.label,
            "Possible raw string key visible: \(element.label)"
        )
    }
}
```

(This heuristic catches `"settings.security.requireAuth"` but not legitimate dot-containing strings like version numbers — refine with a regex if needed.)

#### 3.2 Critical UI elements exist and are non-empty in all languages

Run this same assertion block across all configured locales via the test plan:

```swift
func testCriticalLabelsNonEmptyInCurrentLocale() {
    // Navigate through primary screens and assert key labels are non-empty
    // The test plan runs this in en-US, es-MX, fr-FR, de-DE
    XCTAssertFalse(app.buttons["home.settingsButton"].label.isEmpty)
    XCTAssertFalse(app.buttons["home.editLogsButton"].label.isEmpty)

    app.buttons["home.newLogButton"].tap()
    XCTAssertFalse(app.buttons["logDay.saveButton"].label.isEmpty)
    XCTAssertFalse(app.buttons["logDay.cancelButton"].label.isEmpty)
    app.buttons["logDay.cancelButton"].tap()
}
```

#### 3.3 Date formatting is locale-aware

Assert that a logged date does not appear in hardcoded `"yyyy-MM-dd"` ISO format when the locale is not en-US:

```swift
func testDatesAreNotISO8601InLocalizedBuilds() {
    // Log a day and verify the date shown is not in ISO format
    app.buttons["home.quickLog.noPeriod"].tap()
    // The date displayed on the log card should not match yyyy-MM-dd
    let logCard = app.otherElements["home.dailyLogCard"]
    XCTAssertTrue(logCard.waitForExistence(timeout: 2))
    let isoPattern = #"\d{4}-\d{2}-\d{2}"#
    let labels = app.staticTexts.allElementsBoundByIndex.map { $0.label }
    for label in labels {
        XCTAssertNil(
            label.range(of: isoPattern, options: .regularExpression),
            "ISO date format found in localised build: \(label)"
        )
    }
}
```

#### 3.4 Layout does not overflow under German (longest common translation)

German strings average ~30% longer than English. Assert that all buttons on the log form are still hittable (not truncated off-screen):

```swift
func testLogFormButtonsHittableInAllLocales() {
    app.buttons["home.newLogButton"].tap()
    XCTAssertTrue(app.buttons["logDay.saveButton"].waitForExistence(timeout: 3))
    XCTAssertTrue(app.buttons["logDay.saveButton"].isHittable,
        "Save button is not hittable — may be truncated or off-screen in current locale")
    XCTAssertTrue(app.buttons["logDay.cancelButton"].isHittable)
    app.buttons["logDay.cancelButton"].tap()
}
```

## 6. Static string audit (`make audit-strings`)

Before the string catalog is complete, run a build-time grep that fails if any `Text("…")` with a hardcoded English string still exists in a View file. This is a fast, cheap gate that runs in seconds.

**Script:** `scripts/audit_strings.sh`

```bash
#!/usr/bin/env bash
# Fails if any hardcoded English string literals remain in SwiftUI View files.
# Excludes: empty strings, single characters, SF Symbol names, accessibility identifiers,
# debug-only files, and test files.

set -euo pipefail

SOURCES="Sources/Views"
FAILURES=0

# Pattern: Text("...") where the string is non-empty and not a system name
# Exclude lines that are comments or already use NSLocalizedString / String(localized:)
while IFS= read -r -d '' file; do
    # Skip debug files
    if [[ "$file" == *"Debug"* ]]; then continue; fi

    matches=$(grep -n 'Text("' "$file" | \
        grep -v '//' | \
        grep -v 'Text("")' | \
        grep -v 'accessibilityIdentifier' | \
        grep -v 'accessibilityLabel("' || true)

    if [[ -n "$matches" ]]; then
        echo "HARDCODED STRINGS in $file:"
        echo "$matches"
        FAILURES=$((FAILURES + 1))
    fi
done < <(find "$SOURCES" -name "*.swift" -print0)

if [[ $FAILURES -gt 0 ]]; then
    echo ""
    echo "❌ $FAILURES file(s) contain hardcoded Text() strings."
    echo "   Extract them into Localizable.xcstrings before merging."
    exit 1
fi

echo "✅ No hardcoded Text() strings found."
```

This runs as part of `make audit-strings` and as a pre-merge check in CI.

## 7. Makefile targets

Add the following to `Makefile`:

```makefile
SCHEME     = safeflow
DEST       = platform=iOS Simulator,name=iPhone 16
TESTPLANS  = Tests

## Run all tests (unit + UI, default locale)
test:
	xcodebuild -project safeflow.xcodeproj \
	  -scheme $(SCHEME) \
	  -destination '$(DEST)' \
	  test 2>&1 | xcpretty || xcodebuild -project safeflow.xcodeproj \
	  -scheme $(SCHEME) \
	  -destination '$(DEST)' \
	  test

## Run unit tests only
test-unit:
	xcodebuild -project safeflow.xcodeproj \
	  -scheme $(SCHEME) \
	  -destination '$(DEST)' \
	  -only-testing:safeflowTests \
	  test 2>&1 | xcpretty

## Run accessibility UI tests across all Dynamic Type + a11y configurations
test-a11y:
	xcodebuild -project safeflow.xcodeproj \
	  -scheme $(SCHEME) \
	  -destination '$(DEST)' \
	  -testPlan SafeFlowAccessibility \
	  test 2>&1 | xcpretty

## Run localisation UI tests across en-US, es-MX, fr-FR, de-DE
test-i18n:
	xcodebuild -project safeflow.xcodeproj \
	  -scheme $(SCHEME) \
	  -destination '$(DEST)' \
	  -testPlan SafeFlowLocalisation \
	  test 2>&1 | xcpretty

## Static audit: fail if any View file contains hardcoded Text("...") strings
audit-strings:
	@bash scripts/audit_strings.sh

## Run everything — use before opening a PR
test-all: test-unit test-a11y test-i18n audit-strings
```

`xcpretty` is optional (makes output readable) — the fallback runs plain xcodebuild output if it's not installed.

## 9. CI gate (GitHub Actions or local pre-merge)

Until a formal CI pipeline is configured, the minimum gate is: **`make test-all` must pass before any PR to `release/1.1` is merged.**

The sequence:

```
PR opened to release/1.1
    ↓
make test-unit          ← existing tests pass (regression check)
    ↓
make audit-strings      ← no hardcoded strings remain in Views
    ↓
make test-a11y          ← a11y labels correct; Dynamic Type layouts hold
    ↓
make test-i18n          ← no raw keys; dates locale-aware; layouts hold
    ↓
PR approved and merged
```

If any step fails, the PR is blocked. The error output from `xcodebuild` identifies exactly which test and which screen failed.

## 10. Manual gate (once, pre-submission)

Automated tests catch structural issues (missing labels, broken layouts, raw keys). They do not catch whether VoiceOver's *reading order* is logical, whether hints are helpful, or whether the experience feels right with a screen reader.

One person must do a VoiceOver walkthrough on a **physical device** before the TestFlight build is created:

**Flows to walk:**
1. Onboarding — all 4 pages (swipe through, fill in cycle setup)
2. Home screen — navigate all toolbar buttons, phase card, quick-log
3. Log day form — open, select flow, select symptom, select mood, save
4. Settings — open, toggle auth, close
5. Lock screen (if PIN set)

**Pass criteria:**
- Every interactive element is announced with a meaningful label
- No element is announced as just "button" or "image"
- Tab/swipe order is logical (top-to-bottom, left-to-right)
- Decorative icons are silent (not announced)
- Steppers announce their current value and increment/decrement hints

Document the walkthrough result in a PR comment before merging to `main`.
