# Release Pipeline

**Last updated:** 2026-04-17

---

## Overview

Builds are triggered automatically by git tags pushed to `main`. The tag format determines the destination.

| Tag format | Example | Workflow | Build config | Destination |
|---|---|---|---|---|
| `vX.Y.Z` | `v1.1.1` | `release.yml` | Release | App Store Connect |
| `vX.Y.Z-suffix` | `v1.2.0-beta.1` | `beta.yml` | Beta | TestFlight |

---

## Shipping a release to the App Store

```bash
git checkout main && git pull
git tag v1.1.1
git push origin v1.1.1
```

This triggers `.github/workflows/release.yml` which:
1. Stamps `CFBundleShortVersionString` from the tag (e.g. `1.1.1`)
2. Stamps `CFBundleVersion` from `github.run_number`
3. Builds a Release config IPA using the `safeflow` scheme
4. Uploads to App Store Connect (binary only — metadata and screenshots are managed separately)

Once Apple processes the build (~10–30 min), attach it to the version in App Store Connect and submit for review.

## Shipping a TestFlight beta

```bash
git tag v1.2.0-beta.1
git push origin v1.2.0-beta.1
```

This triggers `.github/workflows/beta.yml` which builds the Beta config and uploads to TestFlight.

## Build configurations

| Config | Swift flags | Debug menu | Test data loader | Signing |
|---|---|---|---|---|
| Debug | `DEBUG` | ✅ | ✅ | Automatic / Development |
| Beta | `BETA` | ✅ | ✅ | Manual / Distribution |
| Release | _(none)_ | ❌ | ❌ | Manual / Distribution |

The Release config is the only one suitable for App Store submission — it strips all debug/beta code paths.

## Metadata and screenshots

Metadata and screenshots are uploaded separately via fastlane, not as part of the build pipeline:

```bash
# Upload all localised metadata and screenshots to App Store Connect
bundle exec fastlane metadata
```

This reads from:
- `fastlane/metadata/<locale>/` — text fields (name, subtitle, description, keywords, release notes)
- `fastlane/screenshots/<locale>/` — PNG screenshots

Screenshots are generated by:
```bash
bundle exec fastlane screenshots
```

Which runs `SnapshotTests` against all 4 locales (en-US, de-DE, fr-FR, es-MX) × 2 device sizes (iPhone 17 Pro Max, iPhone 17).

## Secrets required

All secrets are stored in GitHub repository secrets:

| Secret | Description |
|---|---|
| `CERTIFICATES_P12` | Base64-encoded `.p12` distribution certificate |
| `CERTIFICATES_P12_PASSWORD` | Export password for the `.p12` |
| `PROVISIONING_PROFILE` | Base64-encoded App Store `.mobileprovision` |
| `APPLE_TEAM_ID` | 10-character Apple team ID |
| `ASC_KEY_ID` | App Store Connect API key ID |
| `ASC_ISSUER_ID` | App Store Connect API issuer ID |
| `ASC_KEY_CONTENT` | Contents of the `.p8` API key file (not base64) |

## App Store accessibility declarations

After each release that changes accessibility support, update the declarations in App Store Connect under **App Information → App Accessibility**. Hit **Publish** only after the new version is live.

Current declarations (as of v1.1.1):
- VoiceOver ✅
- Voice Control ✅
- Larger Text ✅
- Dark Interface ✅
- Differentiate Without Color Alone ✅
- Sufficient Contrast ✅
- Reduced Motion ✅

## Known limitations

- **Build number** uses `github.run_number` which resets per-workflow. The release workflow run count is independent of the beta workflow run count. Build numbers must always increase across all uploads to a given app version in ASC — if this causes a rejection, switch to `number_of_commits` as the build number source in both lanes.
- **Territories** are managed manually in App Store Connect — fastlane's `deliver` action no longer supports the `territories` option.
